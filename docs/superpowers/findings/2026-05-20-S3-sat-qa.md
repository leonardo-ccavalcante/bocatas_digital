## Follow-ups — detail (all resolved 2026-05-20b)

### P2-1 — k-anonymity on customQuery aggregate count
**What:** `customQuery.execute` with `aggregate:{op:"count"}` applies no k-anonymity floor, unlike `mapa.distritoStats`. A count of 1 over demographic filters (pais_origen + fase + genero) enables re-identification.
**Why deferred:** customQuery is `adminProcedure`-only and admins legitimately see individual records all day; forcing k-anon would degrade a valid internal analysis tool. The re-id risk is real only when a CSV export is shared *externally* — a process control, not a technical one at the query layer.
**Recommendation:** add an OPTIONAL `kAnonymize: boolean` flag to the spec, default false for internal use, set true when exporting for external/funder consumption. Reuse `K_ANONYMITY_FLOOR` from `server/_core/mapaAggregation.ts`.
**Depends on:** Leo's product decision on internal-vs-external export policy.

### P2-2 — Mapa presence-vs-absence info leak
**What:** A suppressed distrito surfaces as `{distrito:"vicalvaro", count:null}`; a zero-family distrito is absent from `rows` entirely. The difference leaks 1 bit ("≥1 family" vs "0 families") per distrito.
**Why deferred:** low-stakes (operational knowledge of "is X empty" is not sensitive PII), and the choropleth UI needs the null marker to render the polygon neutral.
**Recommendation:** either always-include-all-21-distritos (null for both 0-count and suppressed) or only-include-count≥3 and let the client render absent distritos neutral. ~10 LOC + tests.

### P2-3 — Document the RLS-bypass security posture
**What:** Every tRPC procedure uses `createAdminClient()` (service role) which BYPASSES RLS. The procedure-layer guards (`adminProcedure` + filter clauses) are the actual access boundary; RLS is defense-in-depth for direct DB access only.
**Why it matters:** a new contributor seeing the `report_saved_queries` RLS migration could wrongly trust RLS alone and omit `adminProcedure` on a new procedure → voluntarios get full access. KAC flagged this as the highest-leverage latent trap.
**Recommendation:** add a `## Security model` section to CLAUDE.md or a SECURITY.md stating "service-role bypasses RLS; procedure guards are authoritative; RLS is the safety net."

### P2-4 — Rate-limit `reports.execute`
**What:** No rate limit on the custom-query executor. An admin (or a compromised admin session) can enumerate the dataset via repeated queries.
**Recommendation:** apply the existing rate-limit middleware (see `server/__tests__/rate-limit.test.ts`) to `reports.execute`.

### P2-5 — Audit log for `reports.*` PII access
**What:** Admins can run arbitrary reports against persons/families with no traceability. EIPD/RGPD best practice is to log PII access.
**Recommendation:** emit an audit event (reuse the existing audit pattern in `server/routers/admin/`) on every `reports.execute` and templated-report call, recording `{user_id, entity, filter_count, row_count}` — IDs only, never PII values.

### P2-6 — `String(ctx.user.id)` in savedQueries.list `.or()` filter
**What:** `saved.ts:37` interpolates `${ctx.user.id}` (currently a number, safe) into a PostgREST `.or()` string. `save`/`delete` use `String(ctx.user.id)` consistently; `list` should too, for robustness if the User id type ever changes.
**Recommendation:** 1-line change for consistency.
