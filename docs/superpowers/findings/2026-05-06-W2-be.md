## Findings

### F-W2B-01: N+1 Loop in Offline Sync Batch Insert
**Severity:** CRITICAL  
**Audit:** F-backend  
**File(s):** server/routers/checkin.ts:281–294

**Why:**  
`syncOfflineQueue` loops over input array and awaits one insert per item. On a typical offline flush (10–50 items), this produces 10–50 sequential DB round-trips (~50–250ms) when they could be a single batch insert (< 5ms).

**Fix sketch:**
```typescript
// BEFORE (N+1 loop)
for (const item of input) {
  const { error } = await supabase.from("attendances").insert({...});
  results.push(...);
}

// AFTER (batch insert)
const rows = input.map(item => ({...}));
const { data, error } = await supabase.from("attendances").insert(rows);
if (error && error.code !== "23505") {
  // Handle non-duplicate errors in batch
} else {
  results = input.map((item, i) => ({ clientId: item.clientId, status: "synced" }));
}
```

**Verification:**  
`EXPLAIN ANALYZE` on single `INSERT INTO attendances (...) VALUES (...), (...), ...;` vs. loop. Batch insert should show 1 plan; loop triggers 10+ plans.

**Risk if ignored:**  
Office workers syncing offline queues (end-of-day batch) will see 100–300ms hangs on every 20–50 item flush. Mobile sync becomes unusable during poor connectivity.

### F-W2B-02: Sequential Awaits for Program Enrollment Lookup
**Severity:** HIGH  
**Audit:** F-backend  
**File(s):** server/routers/announcements/reads.ts:38–52 (getAll); lines 138–150 (getById)

**Why:**  
Both procedures fetch `program_enrollments` for user, then fetch `programs` to resolve slugs. Two sequential awaits with zero intervening logic → can parallelize with `Promise.all`.

**Fix sketch:**
```typescript
// BEFORE (sequential)
const { data: enrollments } = await db.from("program_enrollments")...;
const programIds = enrollments?.map(e => e.program_id) ?? [];
if (programIds.length > 0) {
  const { data: programs } = await db.from("programs").in("id", programIds);
  userProgramSlugs = programs?.map(p => p.slug) ?? [];
}

// AFTER (parallel)
const [{ data: enrollments }, { data: programs }] = await Promise.all([
  db.from("program_enrollments").select("program_id")...,
  // OR: conditional Promise.all + lazy fetch if enrollments exist
]);
```

**Verification:**  
Network waterfall in Chrome DevTools. Sequential: 30ms + 17ms = 47ms. Parallel: max(30ms, 17ms) = 30ms.

**Risk if ignored:**  
Dashboard load slows by ~20ms per request. High-traffic deployments see measurable latency increase (cache hit on programs table still requires network round-trip).

### F-W2B-03: Missing Composite Index on announcement_dismissals(person_id, announcement_id)
**Severity:** HIGH  
**Audit:** §B.7 index audit  
**File(s):** supabase/migrations/20260501000005_announcement_dismissals_and_webhook_log.sql  
**Missing:** composite index

**Why:**  
announcements/reads.ts:154 queries `SELECT announcement_id WHERE person_id = ?` to filter visible announcements. Existing `idx_announcement_dismissals_by_person` covers the lookup, but lacks `announcement_id` in the key. Hot path on every announcement list (getAll, getById).

**Fix sketch:**
```sql
CREATE INDEX IF NOT EXISTS idx_announcement_dismissals_person_announcement
  ON announcement_dismissals (person_id, announcement_id)
  WHERE dismissed_at IS NOT NULL;
```

Add to migration **20260501000005** or new phase 6 migration.

**Verification:**  
`EXPLAIN (ANALYZE) SELECT announcement_id FROM announcement_dismissals WHERE person_id = ?;`
- Without composite: Index Scan (person_id partial), then Filter (announcement_id in Filter node).
- With composite: Single Seq/Idx Scan, zero filters.

**Risk if ignored:**  
Dismissal filtering becomes O(M) table scan per visible announcement (M = total dismissals). At 10k+ dismissals, each user sees 2–5ms slowdown on announcement list.

### F-W2B-04: Missing Composite Index on families(estado, deleted_at)
**Severity:** HIGH  
**Audit:** §B.7 index audit  
**File(s):** supabase/migrations/20260411081838_20260410120600_create_families.sql  
**Missing:** composite index

**Why:**  
families/crud.ts:getAll filters `WHERE estado = 'activa' AND deleted_at IS NULL` (lines 49–50). Current indexes cover each individually (`idx_families_estado`, implicit deleted_at), but optimizer may choose suboptimal plan. Composite index ensures tight scan.

**Fix sketch:**
```sql
CREATE INDEX IF NOT EXISTS idx_families_estado_deleted
  ON families (estado, deleted_at)
  WHERE deleted_at IS NULL;
```

Add to **phase 6 migration** (e.g., 20260506000009_family_indexes_v2.sql).

**Verification:**  
`EXPLAIN (ANALYZE) SELECT * FROM families WHERE estado = 'activa' AND deleted_at IS NULL;`
- Without composite: likely sequential scan or double-dip index seeks.
- With composite: single index scan, no filter.

**Risk if ignored:**  
Family list on admin dashboard scans up to 2k rows when filter could reduce to 10 rows via tight index scan. Over 5–10 requests, 100–300ms aggregate latency loss.

### F-W2B-05: Missing staleTime on Read-Only Endpoints
**Severity:** MEDIUM  
**Audit:** F-backend  
**File(s):** Multiple, e.g., server/routers/announcements/reads.ts, server/routers/dashboard.ts, server/routers/checkin.ts

**Why:**  
Read-only tRPC queries (getAll, getById, getKPIStats) have no explicit `staleTime` or React Query cache hints. Default React Query `staleTime` is 0, so every component mount/focus refetches. For stable dashboard data (announcements, KPIs, locations), 30–60s staleTime would eliminate redundant queries.

**Fix sketch:**
```typescript
// In client-side usage (e.g., App.tsx or component mount)
const { data } = trpc.announcements.reads.getAll.useQuery(
  { soloUrgentes: false, limit: 20 },
  { staleTime: 1000 * 60 * 5 } // 5 min for announcements
);

// OR: Server-side cache header (requires tRPC middleware)
export const readsRouter = router({
  getAll: protectedProcedure
    .input(...)
    .query(async ({ input, ctx }) => {
      // Set response header: Cache-Control: public, max-age=300
      return { ... };
    })
});
```

**Verification:**  
React Query DevTools (browser extension). Check "staleTime" column. Should see > 0 for read-only endpoints.

**Risk if ignored:**  
Users see flicker/refetch waterfalls on every page interaction (navigation, focus). Apparent slowness even with fast queries. Bandwidth waste in offline-first scenarios.
