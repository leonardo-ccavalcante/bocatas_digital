## F-04: Recharts Heavy Library in Production Dependencies
**Severity:** HIGH  
**Audit:** C  
**File(s):** package.json:77  
**Why:** CLAUDE.md §7 guard-rail: "Do NOT use heavy chart libraries. Bundle < 300KB enforced by Lighthouse CI." Recharts is ~2.15.4 (estimated 150KB+ gzipped). This violates the bundle constraint and risks Lighthouse ≥95 gate.  
**Fix sketch:** Replace recharts with lightweight alternative (e.g., native SVG sparkline) or defer charts to Gate 2 pending Lighthouse CI validation.  
**Verification:** `npm ls recharts` shows only one entry; rebuild and re-run `npx lighthouse-ci` to confirm bundle stays < 300KB.  
**Risk if ignored:** Lighthouse gate fails at CI; LCP > 2.5s on Moto G breaks Gate 1 acceptance criteria.

## F-05: High-Risk Field (situacion_legal) Exposed in Admin Query Without Role Gating
**Severity:** CRITICAL  
**Audit:** C  
**File(s):** server/routers/persons/crud.ts:144  
**Why:** `getAll` procedure selects `situacion_legal` (high-risk RGPD field per CLAUDE.md §3) for all authenticated users, but uses `protectedProcedure` (any user) instead of `adminProcedure` (admin/superadmin only). Field is supposed to be read-restricted to superadmin/admin by RLS, but this endpoint bypasses RLS by using `createAdminClient()` and does not enforce role check before returning the field.  
**Fix sketch:** Change to `adminProcedure` OR add role gate inside the procedure: `if (!['admin','superadmin'].includes(ctx.user.role)) throw forbidden()`.  
**Verification:** Call `persons.getAll` as volunteer role; confirm `situacion_legal` is NOT in response.  
**Risk if ignored:** RGPD violation: high-risk fields (legal status, document photos, migration history) leak to unprivileged users. Potential data protection authority fine.

## F-06: Direct fetch() to External API Without Auth Validation
**Severity:** MEDIUM  
**Audit:** C  
**File(s):** server/_core/dataApi.ts:31  
**Why:** `callDataApi` uses `fetch()` directly to Forge API endpoint with Bearer token in header. No validation that the token is non-empty or has correct format before use. If `ENV.forgeApiKey` is empty/malformed, request still sends and exposes endpoint.  
**Fix sketch:** Add `if (!ENV.forgeApiKey || ENV.forgeApiKey.length < 20) throw new Error('Invalid API key')` before fetch.  
**Verification:** Set `BUILT_IN_FORGE_API_KEY=""` and try calling `callDataApi()` — should fail at env check, not at network layer.  
**Risk if ignored:** Invalid token exposure in logs; debug noise; potential timing/leakage channel if attacker varies token length and measures response time.

## F-07: Consent Templates Missing Spanish Language Seed
**Severity:** CRITICAL  
**Audit:** I  
**File(s):** supabase/migrations/EXPORTED/20260413121730_20260501100700_seed_consent_templates_bda.sql  
**Why:** CLAUDE.md §3 mandate: "Consent is multi-language from day 1. Required languages: Spanish, Arabic, French, Bambara." Seed migration only inserts ar, fr, bm (lines 5-16); Spanish ('es') is missing. Beneficiaries cannot consent in their primary language.  
**Fix sketch:** Add INSERT for idioma='es' with Spanish text before `ON CONFLICT` clause. Example: `('tratamiento_datos_banco_alimentos', 'es', '1.0', 'Autorizo el tratamiento de mis datos personales...', true),`  
**Verification:** SELECT * FROM consent_templates WHERE idioma='es' AND purpose='tratamiento_datos_banco_alimentos' — must return 1 row.  
**Risk if ignored:** Non-Spanish speakers cannot read/sign consent; legal challenge to RGPD compliance; audit finding for data protection authority; Gate 1 cannot launch.
