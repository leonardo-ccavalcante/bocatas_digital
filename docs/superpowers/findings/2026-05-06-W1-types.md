## Findings (Dimension A: TypeScript Strictness)

## F-1: Unsafe `as unknown as` cast in CSV export without comment
**Severity:** HIGH  
**Audit:** A  
**File(s):** server/csvExport.ts:114, server/csvExportWithMembers.ts:157, server/csvExportWithMembers.ts:176, server/csvExportWithMembers.ts:182  
**Why:** Dynamic field access via chained type assertion bypasses type safety; no comment explains why the cast is necessary instead of using a proper schema.  
**Fix sketch:** Either import/use a Zod schema for family/member shape, or add a one-line comment explaining why the cast cannot be avoided (e.g., "Supabase RLS returns untyped row; shape validated upstream").  
**Verification:** `rg 'as unknown as Record<string, unknown>' server/csvExport*.ts --type ts` should show zero occurrences or all with explanatory comments.  
**Risk if ignored:** Silent type errors if family or member field structure changes; data corruption during CSV export if schema assumption breaks.

## F-2: Unchecked `as unknown as` in bulkimport preview without comment (3 cases)
**Severity:** HIGH  
**Audit:** A  
**File(s):** server/routers/announcements/bulk-import.ts:114, :175, :195  
**Why:** Bulk import preview data is stored as JSONB and cast without documented type safety or narrowing.  
**Fix sketch:** Add a comment above the cast explaining why the preview data cannot be pre-typed (e.g., "Supabase stores JSONB without schema; type valid on retrieval via parser validation").  
**Verification:** Read the cast sites; verify each has a one-line justification within 2 lines above.  
**Risk if ignored:** Bulk announcements could silently fail or import malformed data if shape contract is violated.

## F-3: Unchecked `as unknown as` in compliance reporting (missing comments)
**Severity:** HIGH  
**Audit:** A  
**File(s):** server/routers/families/compliance.ts:166, :228  
**Why:** Missing document lists are cast to `string[]` without comment; no validation that the values are actually strings.  
**Fix sketch:** Add comment: "familyMissingDocs is string[] per schema validation; cast narrows unknown RLS result."  
**Verification:** `grep -A1 -B1 'missing: familyMissingDocs as unknown' server/routers/families/compliance.ts` should show a justification comment.  
**Risk if ignored:** Noncompliance audit data could contain non-string values, causing downstream report rendering failures.

## F-4: Unsafe clearTimeout type assertion in QRScanner (type mismatch)
**Severity:** MEDIUM  
**Audit:** A  
**File(s):** client/src/features/checkin/components/QRScanner.tsx:70, :76  
**Why:** `setTimeout` returns `NodeJS.Timeout | number` (environment-dependent), but code conflates it with `number` for both timeout and raf IDs; the double cast `as unknown as number` masks a real runtime bug.  
**Fix sketch:** Use separate refs: `timeoutRef: React.MutableRefObject<ReturnType<typeof setTimeout> | null>` and `rafRef: React.MutableRefObject<number | null>`. Cast only at platform boundary.  
**Verification:** Run in Node and browser; verify `rafRef.current` is always a number and `timeoutRef.current` is always a Timeout, with no implicit coercion.  
**Risk if ignored:** clearTimeout/cancelAnimationFrame calls may fail silently on platform transition or in environments where types differ (e.g., jsdom).

## F-5: 48 unexamined eslint-disable @typescript-eslint/no-explicit-any in server code
**Severity:** MEDIUM  
**Audit:** A  
**File(s):** server/routers/csv-export.ts:7,9,33,35; server/routers/dashboard.ts:75,125,188,251; server/routers/programs.ts:82,272,332; server/routers/families/csv-import.ts:79,93,104,164,189; server/routers/families/members.ts:92; server/routers/families/crud.ts:214; server/routers/families/compliance.ts:290; server/routers/families/documents.ts:36,195,205; server/routers/families/sessions.ts:21; server/routers/families-csv-members-procedures.ts:36 (partial list; 48 total)  
**Why:** Bulk of server adapter code systematically escapes type checking for Supabase RLS results and query builder chains (`.eq()`, `.filter()` return `any`); no unifying pattern or comment explaining why (e.g., "Supabase PostgREST types unavailable at this depth").  
**Fix sketch:** File one group finding: consolidate into a typed Supabase client wrapper that narrows RLS results once at the edge, then remove disable comments in adapters.  
**Verification:** After wrapper is written, run `rg 'eslint-disable.*no-explicit-any' server/routers | wc -l` → should drop to <5 (for unavoidable cases).  
**Risk if ignored:** RLS-touching code remains untypeable; vulnerability drift invisible to type checker.

## F-6: Record<string, unknown> pattern in dynamic field access without schema validation
**Severity:** MEDIUM  
**Audit:** A  
**File(s):** server/csvExport.ts:114; server/csvExportWithMembers.ts:157,176,182; server/routers/announcements/crud.ts:132; server/routers/families/crud.ts:298; server/routers/ocr.ts:205; client/src/pages/NovedadDetalle.tsx:111,112; client/src/pages/FamiliaDetalle/index.tsx:243  
**Why:** Dynamic object indexing is common pattern (8+ files) to handle untyped RLS results; each site repeats the cast instead of using a reusable typed schema or utility.  
**Fix sketch:** Create a utility `extractField(obj: unknown, key: string, defaultValue?: any): unknown` that documents the assumption once. Or define explicit types in Zod for families/members and validate at read.  
**Verification:** Confirm each call to `Record<string, unknown>` is guarded by upstream schema validation (Zod parse or RLS type narrowing).  
**Risk if ignored:** Silent null/undefined access errors; scattered maintenance burden.

## Findings (Dimension D: Coding Standards)

## F-7: Client console.error calls without user-data checks (10 instances)
**Severity:** LOW  
**Audit:** D  
**File(s):** client/src/main.tsx:28,36; client/src/components/MemberManagementModal.tsx:171; client/src/components/ImportFamiliesModal.tsx:98; client/src/components/AnnouncementImageUploader.tsx:48; client/src/components/DeliveryDocumentModal.tsx:73; client/src/components/DeliveryDocumentUpload/_useOcrFlow.ts:110; client/src/components/DeliveryDocumentUpload/_useCsvFlow.ts:45; client/src/pages/AdminNovedades/index.tsx:237; client/src/pages/FamiliasList.tsx:78  
**Why:** 10 console error/log calls remain in client production code; only one (`AdminNovedades:237`) logs user data (`console.log(Reschedule ... to ...)`) which is low-severity.  
**Fix sketch:** Move all console calls to a logger utility (e.g., `shared/logger.ts`) that respects a log level env var (disabled in prod). Keep error boundaries for user-facing errors.  
**Verification:** `rg 'console\.(log|error|warn|info)' client/src --type ts --type tsx | wc -l` → should be 0 or <2 (for logging wrapper exports).  
**Risk if ignored:** Performance / privacy leak in production if user data appears in logs; inconsistent error handling.

## F-8: eslint-disable without comment on react-hooks/exhaustive-deps (4 instances)
**Severity:** MEDIUM  
**Audit:** D
